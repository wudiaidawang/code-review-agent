# AI Code Intelligence Platform — V1 完整任务计划书

> **状态：重构版（2026-07-11）**
> **目标：交付一个可验证、可解释、可扩展的 AI 代码智能分析平台核心，而不是堆叠 Agent 概念的 Code Review Demo。**

---

## 一、项目定位与边界

### 一句话定位

> 面向代码变更审查和代码库探索的 AI Code Intelligence Platform：确定性工具负责提取可验证事实，LLM 只负责基于事实做语义推理与解释。

### V1 必须证明的能力

1. 对 Python 代码变更生成可追溯的结构化审查结果。
2. 每一条 Issue 都能定位到变更、代码符号或工具证据，避免无依据的 LLM 结论。
3. 静态分析、AST、依赖分析等工具可被固定 Review Pipeline 和后续 Investigation Agent 共同使用。
4. 系统可用离线评测集客观衡量检出效果、误报、耗时和 token 成本。
5. 小模型（微调）只负责受约束的审查计划生成；高能力模型负责复杂语义判断。

### V1 明确不做

- 不把 Code Review 包装为 Agent；Review 使用确定性 Pipeline。
- 不在工具层强行输出 Issue；工具首先输出事实与证据。
- 不以多角色 Prompt 数量、向量库或 LangGraph 作为项目亮点。
- 不优先开发大而全前端、GitHub 自动评论、指标大盘或无限制 Agent。
- 不执行待分析仓库中的不可信代码；V1 只读取、解析和调用受控静态工具。

## 二、不可妥协的设计原则

| 原则 | 落地约束 |
|---|---|
| 工程可靠性优先 | 每个工具有明确输入、输出、超时、失败状态；Pipeline 可复现。 |
| 确定性优先 | 能由 Git、AST、Ruff、Bandit、依赖图回答的问题，不调用 LLM。 |
| LLM 基于证据推理 | LLM 输入必须包含变更和证据；输出必须绑定文件、行号/符号与证据 ID。 |
| 工具与编排解耦 | 工具不依赖 Pipeline 或 Agent；两种编排仅消费统一工具契约。 |
| 可评测而非“看起来聪明” | 每个阶段都定义功能、质量、成本和回归验收。 |
| 渐进扩展 | 新语言、新 Analyzer、新策略优先新增模块，不修改已有执行内核。 |

## 三、目标架构

```text
请求（PR / 本地仓库 / 调查问题）
             │
             ▼
Repository Workspace（受控 clone / worktree / 资源限制）
             │
             ▼
Tool Layer（统一契约、无编排依赖）
Git Diff · AST/Symbol · Search · Ruff · Bandit · Dependency · RAG · LLM
             │
             ▼
Artifact & Evidence Layer
ChangeSet · SymbolIndex · ToolFinding · DependencyGraph · KnowledgeEvidence
             │
     ┌───────┴────────┐
     ▼                ▼
Review Mode       Investigation Mode（V1.1）
固定 ReviewPlan   受限工具选择、证据累积、引用式回答
     │                │
     └───────┬────────┘
             ▼
LLM Semantic Reasoning（仅在需要时）
             │
             ▼
Issue / Answer + Evidence References + Execution Trace
```

### 3.1 核心对象与职责

| 对象 | 职责 | 不承担的职责 |
|---|---|---|
| `Tool` | 输入校验、执行具体能力、返回结构化 `ToolResult` | 决定调用顺序、生成最终报告 |
| `Artifact` | 保存可复用的事实产物 | 承载业务结论 |
| `Evidence` | 可引用的事实片段，含来源、路径、行号/符号、置信度 | 替 LLM 做推理 |
| `Finding` | 工具或规则产生的候选发现 | 统一对用户展示的最终问题 |
| `Issue` | 经聚合、去重、校验后的审查问题 | 记录原始工具数据 |
| `ReviewPlan` | 明确本次调用哪些 Analyzer、顺序、预算与理由 | 执行工具实现 |
| `ReviewRun` | 本次输入、产物引用、执行 trace、统计、失败信息 | 充当无边界数据字典 |

### 3.2 工具统一契约

```python
class Tool(Protocol):
    name: str

    def execute(self, request: ToolRequest) -> ToolResult:
        """返回 artifacts、evidence、findings、diagnostics 与耗时；不直接修改上下文。"""
```

工具输出不必统一成 `Issue`：例如 AST 返回符号索引，Git 返回变更集，Ruff/Bandit 返回可映射的规则发现。仅在 Aggregation 阶段统一为 `Issue`。

### 3.3 Review Pipeline

```text
请求校验 → 获取变更 → 生成事实产物 → 生成 ReviewPlan
        → 执行计划中的确定性 Analyzer → 聚合候选发现
        →（必要时）LLM 语义审查 → 证据校验/去重/分级 → 报告
```

Pipeline 默认使用规则策略生成计划；所有运行都保存 `ReviewPlan` 与执行 trace，保证可解释和可复现。

### 3.4 Investigation Agent（V1.1，不阻塞 V1）

它不是第二套分析系统，而是同一工具层上的受限编排器：只可选择注册工具、必须在预算和步数内运行、每次回答必须返回引用的 `Evidence`。没有稳定 Artifact/Evidence 层之前，不实现 Agent。

## 四、V1 范围与里程碑

V1 以 Python、Git 变更审查为唯一垂直切片。每个里程碑都必须可独立演示和回归验证。

### M0：领域模型与受控工作区

**目标：** 定义长期稳定的分析契约，并安全地准备目标仓库。

- 定义 `ChangeSet`、`FileChange`、`CodeLocation`、`Symbol`、`Evidence`、`Finding`、`Issue`、`ReviewPlan`、`ReviewRun`。
- 将现有 `ReviewContext` 从“大公文包”收敛为运行元数据和产物索引；保留兼容迁移路径。
- 建立本地仓库/worktree 管理：路径白名单、大小/文件数限制、超时、清理策略、只读分析约束。
- 定义所有工具的 `ToolRequest` / `ToolResult`、错误代码和 telemetry 字段。

**验收：**

- 可构造和序列化完整 `ReviewRun`，其所有 Issue 可反向追溯 Evidence。
- 工具失败不会吞异常：返回结构化诊断，不使后续可运行步骤失效。
- 不可信仓库不会被导入或执行。

### M1：确定性事实层

**目标：** 不调用 LLM，也能得到可信的变更、代码结构和静态规则事实。

- `GitTool`：解析 base/head 或本地提交范围，输出结构化 `ChangeSet` 与 hunk/行号映射。
- `PythonParserTool`：以 Tree-sitter 或 Python AST 实现符号提取；输出函数、类、导入与调用边。
- `RuffTool` / `BanditTool`：只分析变更关联文件；输出原始诊断和标准化 `Finding`。
- `DependencyTool`：基于导入与符号调用输出有限、可解释的影响范围；不承诺不可靠的全局精确调用图。
- `SearchTool`：提供只读、可限制范围的代码/符号搜索。

**验收：**

- 测试仓库覆盖新增、修改、删除、重命名文件及空 diff。
- 每一个静态 Finding 带规则来源、代码位置和对应 Evidence。
- 针对变更文件运行分析，且不执行目标项目代码。
- 纯确定性测试可离线运行。

### M2：固定 ReviewPlan 与高可信审查闭环

**目标：** 用固定、透明的计划完成一次端到端审查。

- 实现规则式 `ReviewPlanBuilder`：基于语言、文件类型、diff 规模与风险信号选择工具。
- 引入最低安全策略：认证、权限、SQL、命令执行、反序列化、依赖文件等风险信号不能被策略跳过。
- 实现 `ReviewExecutor`：执行计划、记录 trace、超时/失败降级。
- 实现 Aggregator：基于位置、规则类型与语义标签去重；保留全部来源与证据。
- 实现确定性 Markdown/JSON 报告；评分仅作为辅助风险信号，不应伪装成客观质量分。

**验收：**

- 同一个输入、同一工具版本与配置得到相同的确定性结果和计划。
- 报告展示：变更摘要、执行/跳过原因、每个 Issue 的证据和来源、未完成步骤。
- 规则问题不会因 LLM/RAG 不可用而丢失。

### M3：LLM 语义审查与知识证据

**目标：** 只把静态工具无法可靠判定的语义问题交给 LLM。

- 制定结构化 LLM 请求：最小必要 diff、相关符号、静态发现摘要、影响范围和知识证据。
- LLM 输出必须含 `location`、`reason`、`suggestion`、`evidence_ids`；不满足 schema 或无法关联证据则标记为低置信度候选，不直接生成高严重度 Issue。
- 将知识库从“启动即加载模型的三 collection”改为可插拔 `KnowledgeRetriever`；知识条目必须带来源、版本、许可/引用信息。
- RAG 仅在策略判断需要规范依据时调用；检索失败降级为无外部规范的审查，不阻塞 Pipeline。
- 完成 LLM 输出解析、重试、预算、分块和缓存策略。

**验收：**

- LLM 结论可回链到 diff、符号或知识证据。
- JSON/schema 异常、服务失败与超限均有可观测降级结果。
- 在评测集上记录 LLM 增量检出、误报、耗时与 token 成本，而非只展示单个成功案例。

### M4：评测体系与微调 Review Planner

**目标：** 证明“小模型做决策、大模型做推理”能带来可量化价值。

#### 微调任务定义

输入：变更摘要、文件类型、diff 规模、风险特征、AST 摘要、已有确定性发现。
输出：受 schema 约束的 `ReviewPlan`：Analyzer 集合、执行优先级、是否调用 RAG/LLM、预算、理由标签。

```json
{
  "analyzers": ["git", "python_ast", "ruff", "bandit", "dependency"],
  "enable_rag": false,
  "enable_llm_semantic_review": true,
  "risk_level": "high",
  "reason_codes": ["auth_change", "tainted_input", "public_api_change"]
}
```

#### 数据构建

- 从评测 PR/提交及规则式策略生成初始计划。
- 由人工审阅修订高价值样本，特别标记“遗漏会造成风险”的 Analyzer。
- 利用真实执行 trace、最终检出结果与成本形成偏好/监督数据。
- 训练、验证、测试按仓库或时间切分，避免同一模式泄漏。

#### 客观指标

- Analyzer 选择 precision / recall / F1；高风险必选工具召回率必须单列。
- 相对于规则式基线的平均耗时、token、工具调用数。
- 下游 Issue 检出率、人工确认误报率、严重 Issue 漏报率。
- 输出 schema 合法率和策略安全兜底触发率。

#### 上线策略

- 规则式 Planner 永远是可回退基线。
- 高风险安全规则优先于模型预测，模型不可跳过。
- 先 shadow mode 比较预测计划与基线，再允许受控启用。

**验收：**

- 有版本化数据集、基线、模型、评测脚本与报告。
- 微调 Planner 至少在成本或工具选择质量一项上显著优于基线，且不降低高风险工具召回率。
- 若未达标，保留规则式 Planner；微调实验仍作为可复现实验结论，而不是强行上线。

### M5：最小服务化与演示

**目标：** 让核心能力可使用，但不让 UI 反向驱动架构。

- FastAPI：提交 Review 请求、查询 `ReviewRun`、获取结构化报告。
- 最小前端或 CLI：展示变更、策略 trace、Issue 及证据跳转。
- 本地持久化：保存 run、plan、Issue、Evidence 引用和评测元数据。
- Docker/README/示例仓库：确保其他人可以复现一个演示和一套测试。

**验收：**

- 从仓库/提交范围发起审查并查看带证据的结果。
- API 错误采用统一 schema，运行可查询，敏感配置不入库。
- 演示以评测样本为主，不只依赖手工构造的成功案例。

### M0—M5 的实施顺序、阶段交付与验收指标

不要按架构图逐层“写完所有模块”再联调；应按下表构建连续的能力闭环。每完成一个阶段，系统都必须可运行、可测试，并成为后续阶段唯一依赖的稳定基础。

| 实施阶段 | 基于什么继续做 | 本阶段新增交付 | 完成后系统具备的能力 | 必须通过的验收标准 |
|---|---|---|---|---|
| 阶段 0：现有骨架 | 现有 `Issue`、`ReviewContext`、`Pipeline` 与 LLM/RAG 原型 | 清理旧原型的职责边界，建立测试基线 | 可安全重构且不丢失已有行为 | 当前单元测试全部通过；旧模块不再作为后续架构依赖 |
| 阶段 1：可追溯数据契约（M0） | 阶段 0 的骨架 | `ChangeSet`、`Evidence`、`Finding`、`ReviewPlan`、`ReviewRun` 与 Tool 契约 | 能表达一次审查的输入、证据、计划、结论和失败状态 | 每个 `Issue` 至少可关联一条 Evidence；模型序列化、错误状态和引用关系均有单测 |
| 阶段 2：确定性事实层（M1） | 阶段 1 的契约 | Git Diff、Python AST、Ruff、Bandit、依赖/搜索工具 | 输入本地 Git 提交范围，输出结构化变更、符号和静态发现 | fixture 覆盖新增/修改/删除/重命名/空 diff；静态 Finding 100% 带路径、行号与规则 ID；不执行目标仓库代码 |
| 阶段 3：固定审查闭环（M2） | 阶段 2 的事实产物 | 规则式 `ReviewPlanBuilder`、Executor、初步 Aggregator、Markdown/JSON 报告 | 对真实 Python 变更生成可解释、可复现的审查报告 | 相同输入、相同工具版本得到相同计划与确定性结果；报告展示执行/跳过项及理由；端到端集成测试通过 |
| 阶段 4：LLM 增量语义审查（M3） | 阶段 3 的闭环与证据 | 风险驱动 LLM 调用、RAG、最终 Aggregator 与 Evidence Validator | 发现静态工具难以可靠判断的逻辑/设计风险 | LLM 输出 schema 合法；LLM Issue 的证据绑定率为 100%；LLM/RAG 失败时仍返回确定性结果 |
| 阶段 5：评测与微调 Planner（M4） | 阶段 4 的运行 trace 与结果 | 标注评测集、规则基线、微调数据、Planner 实验与评测报告 | 小模型选择分析工具，大模型仅处理语义推理 | 报告 Analyzer F1、高风险工具召回、检出率、误报、耗时和 token；与规则基线比较，且不降低高风险工具召回 |
| 阶段 6：服务化与演示（M5） | 阶段 5 的稳定内核 | FastAPI、CLI/最小前端、持久化、Docker、Demo | 他人可发起审查、查看 Evidence 与执行 trace | 一条命令启动；至少 3 个固定评测 Demo；API、容器与评测复现流程通过 |

**推进门槛：** 阶段 3 是第一个可独立展示、可写入简历的版本。阶段 5 是项目最具差异化的版本；在阶段 5 之前，不启动 Investigation Agent、GitHub PR 回写或复杂前端工作。

### 指标口径与简历表达

简历只能填写真实跑出的结果，不预填数字。每个指标须记录数据集版本、工具/模型版本、运行日期和计算脚本，以便复现。

| 指标 | 计算口径 | 对应阶段 | 简历表达模板 |
|---|---|---|---|
| 证据可追溯率 | 带至少一个有效 Evidence 引用的最终 Issue / 最终 Issue 总数 | 阶段 1 起 | “构建 Evidence 引用链，审查结论可追溯率达到 `X%`。” |
| 静态定位完整率 | 同时带文件路径、行号、规则 ID 的静态 Finding / 静态 Finding 总数 | 阶段 2 | “集成 Git Diff、AST、Ruff、Bandit，静态发现定位完整率达到 `X%`。” |
| Pipeline 可复现率 | 相同输入与版本下，计划及确定性 Issue 完全一致的运行数 / 总运行数 | 阶段 3 | “实现可复现 Review Pipeline，在固定评测集上确定性结果一致率达到 `X%`。” |
| LLM 证据约束率 | 带有效 Evidence 引用的 LLM Issue / LLM Issue 总数 | 阶段 4 | “采用证据约束的 LLM 语义审查，LLM 结论证据绑定率达到 `X%`。” |
| Planner 工具选择质量 | 相对人工/标注计划的 Analyzer precision、recall、F1；安全工具单列 recall | 阶段 5 | “微调 Review Planner，Analyzer 选择 F1 达到 `X`，高风险安全工具召回率为 `X%`。” |
| 成本与性能 | 每次审查的平均/分位耗时、token、工具调用数；相对规则基线变化 | 阶段 5 | “在不降低高风险工具召回的前提下，将平均 token/工具调用/审查耗时降低 `X%`。” |
| 下游审查质量 | 标注集 Issue 的 precision、recall、严重问题漏报率 | 阶段 4—5 | “在 `N` 个标注变更集上评测，Issue precision/recall 为 `X/X`，严重问题漏报率为 `X%`。” |
| 工程质量 | 单元、合约、集成、评测测试数量及通过率 | 全程 | “构建 `N` 条单元、合约与集成测试，并提供可复现评测与容器化演示。” |

每阶段提交前至少执行对应单元/集成测试；每个阶段结束时保存一份简短的阶段报告，记录已完成能力、失败样本、指标、已知限制和下一阶段输入。这样最终的简历指标来自持续积累的实验记录，而不是收尾时临时补测。

## 五、ReviewPlan：规则基线与模型演进

### 5.1 规则式 V1 基线

| 信号 | 必须/推荐动作 |
|---|---|
| `.py` 变更 | Git、Python AST、Ruff；按风险决定 Bandit/依赖/LLM。 |
| 认证、授权、输入校验、SQL、命令、反序列化 | 强制 Bandit + 安全规则 + LLM 安全语义审查。 |
| 公共函数、模型基类、配置、核心模块 | 增加依赖影响分析；提升审查预算。 |
| 文档/纯文本 | 记录跳过理由，不调用 LLM。 |
| 依赖清单、Dockerfile、CI 配置 | 走专用安全/配置 Analyzer；不套 Python AST。 |
| 大 diff | 按文件/符号分块，限制每块 token 和总预算。 |

### 5.2 为什么先做基线再微调

微调模型的标签、指标和安全下限都依赖一个可执行的策略基线与真实 trace。没有基线时，模型只是在模仿主观 Prompt；有基线后，模型才是在优化可衡量的“覆盖率—成本—延迟”权衡。

## 六、质量、安全与可观测性

### 6.1 安全边界

- clone/worktree 必须限制目标目录、仓库大小、文件数和执行时间。
- 不 import、不运行、不安装目标仓库依赖；静态工具使用受控命令与参数。
- 敏感配置只从环境变量/密钥系统读取；报告默认脱敏 Token、密码和个人信息。
- 外部 GitHub 回写属于后续集成能力，必须显式授权、dry-run 和幂等。

### 6.2 运行记录

每次 `ReviewRun` 至少保存：输入版本、工具版本、ReviewPlan、每步状态/耗时、失败诊断、token 使用、Artifacts/Evidence/Issues 的引用关系。

### 6.3 测试分层

| 层级 | 内容 |
|---|---|
| 单元测试 | 数据模型、diff 解析、AST 提取、规则映射、聚合、计划选择。 |
| 合约测试 | 每个 Tool 的请求/结果 schema、错误语义、超时处理。 |
| 集成测试 | 小型 Git fixture 仓库上的固定 Pipeline。 |
| 评测测试 | 标注变更集的检出、误报、成本、延迟回归。 |
| LLM 测试 | mock/schema/引用校验为主；真实模型跑固定 benchmark，不作为单测依赖。 |

## 七、当前仓库迁移策略

当前仓库已有 `Issue`、`ReviewContext`、`Pipeline` 骨架，以及旧版单文件 LLM/RAG 审查模块；尚未实现 Git Diff、AST、Ruff/Bandit、端到端 Review Pipeline。

迁移顺序：

1. 不继续扩大旧 LangGraph/“多 Reviewer Prompt”路径。
2. 保留 `Issue` 的兼容字段，但新增 Evidence/Artifact/ReviewPlan 模型；逐步替换 `ReviewContext` 的无类型占位字段。
3. 将 `llm_tool`、知识检索与静态工具适配为统一 Tool 契约，不让它们依赖 Pipeline 内部状态。
4. 在 M1/M2 完成前，将现有 Chroma 种子视为演示数据，而非平台核心能力。
5. 每一阶段完成后更新 `docs/INDEX.md`；用户执行 push 前，将该段所有改动完整补入 `docs/CHANGELOG.md`。

## 九、成功标准

V1 成功不以“调用了多少模型、接了多少工具、是否有 Agent”衡量，而以以下问题回答“是”为标准：

1. 面对一个真实变更，系统是否能说明发现了什么、依据是什么、哪些分析已执行或被跳过、为什么？
2. 静态问题、LLM 语义问题和未知/失败状态是否被诚实区分？
3. 添加一个 Analyzer 是否只需实现 Tool/Adapter 并注册到 Planner，而不修改 Pipeline 内核？
4. 是否能用评测数据证明策略、LLM 或微调模型带来了检出质量、成本或延迟的改进？
5. 当 LLM、RAG 或某个工具失败时，系统是否仍能安全、可解释地返回已验证结果？

若上述标准成立，这个项目展示的是 AI 工程系统设计能力，而不是一次性代码审查演示。

### 各层用处速查

| 层 | 用处（该层缺失会坏在哪） |
|---|---|
| ① 入口 | 把 PR / 本地 / 调查质问三种触发源正规化为一件事，避免下游到处分叉 |
| ② Workspace | 安全境界；不执行目标代码，杜绝"跑了不可信仓库"的风险 |
| ③ Tool Layer | 工具与编排解耦；Review 与 Investigation 复用同一批工具，是 V1 扩展性的核心 |
| ④ Artifact/Evidence | 事实沉淀；每条结论都绑定 Evidence（来源+行号+置信度），杜绝无依据的 LLM 结论 |
| ⑤A ReviewPlan | 规则基线与微调候选计划共同生成，经过安全约束校验；高风险工具不能被模型跳过 |
| ⑤B Investigation | V1.1 受限 Agent；Artifact 层稳定前不实现 |
| ⑥ LLM 语义 | 只做基于证据的语义推理；静态工具能定的不给 LLM |
| ⑦ 最终聚合 | LLM 与确定性发现统一去重、校验并标记置信度，避免 LLM 绕过质量门槛 |
| ⑧ 输出 | Issue + 证据引用 + trace，做到可解释、可复现、可评测 |
| ⑨ 服务层 | 最小 API/CLI 薄壳，防止 UI 反向决定架构 |
| ⑩ 评测 | 用数字而非"看起来聪明"来验证策略/LLM/微调是否真的带来改进 |
