# 项目编码规范（长期生效）

本文件是本项目的最高优先级协作约定。任何会话、任何模型在本仓库工作时都必须遵守以下规范。

## 规范零：输出语言

**面向用户的所有输出一律使用中文。** 用户是中文母语者，任何解释、总结、报告、提问、回答都必须用中文书写（代码、命令、标识符、专有名词等技术符号本身除外）。此规则优先级最高，任何会话、任何模型都不得用其他语言回复用户。

## 规范一：文件索引管理

`docs/INDEX.md` 是全项目的文件索引，逐文件概括其**结构**与**作用**。

- **新建文件时**：必须在 `docs/INDEX.md` 追加该文件的条目（路径 + 一句话作用 + 关键结构/函数/类概览）。
- **修改文件、导致其作用或结构发生变化时**：必须同步更新 `docs/INDEX.md` 中对应条目。
- **删除/重命名/移动文件时**：必须同步修正或移除 `docs/INDEX.md` 中对应条目。
- 目的：让后续任何人（包括未来的会话）无需到处翻找，读 INDEX 即可掌握代码分布与职责。

## 规范二：修改留痕（CHANGELOG）

`docs/CHANGELOG.md` 记录所有代码改动，用于回溯与查证。核心原则：**任何修改都必须记录在案，绝不允许「为了 A 改了 B，却隐瞒不报 B」的情况。**

- **形式**：CHANGELOG，以「一段工作时长」为记录单位（而非每次文件保存、也非每个 git commit）。
- **时机**：在用户执行 `git push` 时补写——即 push 前，把本段工作期间的所有改动整理成一条 CHANGELOG 记录再提交。
- **每条记录必须包含**：
  - **日期**（绝对日期，如 2026-07-09）
  - **改了什么**：涉及的文件与具体改动点，**逐一列全，一个都不能漏**。
  - **为什么改**：动机 / 触发原因。
  - **A → B 因果**：若「为了实现 A 而不得不动了 B」，必须显式写出这条因果链，禁止只报 A 隐瞒 B。
- 记录持久化在 `docs/CHANGELOG.md`，纳入 git 版本管理，随代码提交，可长期回溯。

## 执行说明

以上两条不依赖任何自动 hook，由在本仓库工作的助手自觉遵守。开始任何文件改动前，先意识到这两条约束；在 push 前，务必完成 INDEX 与 CHANGELOG 的同步。
