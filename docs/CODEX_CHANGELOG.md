# 企业化加固变更日志

本日志只记录账号、数据隔离、写入一致性与前端身份交互改动；不替代项目总 [CHANGELOG](CHANGELOG.md)。

## 2026-07-24 — 前端固定 Review Pipeline（审查管道）模式

- 前端会话输入区新增“代码调查 / 变更审查”模式切换；审查模式可填写 `base_ref`（基准 Git 版本）与 `head_ref`（目标 Git 版本）。
- 审查请求提交到受登录用户保护的 `/jobs/review`，只发送 `repo_id`（仓库标识），后端按当前用户解析真实仓库路径。
- 审查结果沿用异步 Job（任务）与 SSE（服务端事件流），在聊天区展示 Markdown（报告）、Issue（问题）与 Evidence（证据）诊断。
- 固定审查管道仍为 `PlanBuilder（计划生成器）→ Executor（执行器）→ Aggregator（聚合器）→ Report（报告生成器）`；它与 Investigation（代码调查）模式分开，避免自然语言问答污染规则型变更审查。

## 2026-07-24 — SQLite（轻量数据库）一致性与账号边界加固

### 风险与改动

| 风险 | 改动 | 效果 |
|---|---|---|
| SQLite 多请求写入等待/锁错误 | 开启 WAL（预写日志）、`busy_timeout=10s`、外键与显式事务 | 读写并发更稳定，短暂锁竞争会等待而非立刻失败 |
| 两个页面保存同一历史导致后写覆盖先写 | `conversations` 增加 `version`（乐观锁版本），更新/删除要求版本匹配 | 冲突返回 HTTP 409，不发生静默丢失 |
| 不同用户恰好使用同一客户端对话 ID | 主键从全局 `id` 改为 `(user_id, id)` 复合主键 | 用户命名空间独立，查询/更新/删除始终带 user_id |
| 已有本地库无法升级 | 首次启动检测旧表并在 `BEGIN IMMEDIATE`（立即写事务）中迁移 | 升级过程原子化，保留既有历史 |
| 明文密码、可长期使用的会话 | scrypt（密码哈希）+ 随机 salt（盐值）、仅存 token hash（令牌哈希）、14 天过期与登出撤销 | 数据库泄露时不暴露明文密码/原始令牌 |
| 历史消息无限写入 | 限制对话数据信息：最多 200 条消息、单消息正文最多 16000 字符 | 控制本地数据库膨胀与异常 payload（载荷） |
| 浏览器退出只清本地令牌 | 前端调用 `/auth/logout` 撤销服务端 session（会话） | 令牌不能继续被服务端接受 |

### 前端改动

- 登录/注册界面在未认证前遮挡调查页面；登录后才加载当前账号的历史。
- 所有历史读写携带 Bearer Token（持有者令牌）。
- 前端保存成功后使用服务端返回的最新 `version`；冲突时保留本地页面内容并提示刷新，不盲目重试。
- 删除历史携带 `X-Conversation-Version`（对话版本请求头），防止删除他人页面刚更新的记录。
- 退出会先调用服务端令牌撤销接口，再删除浏览器本地令牌。

### 后端改动

- `app/persistence/users.py`：事务、WAL、迁移、乐观锁、口令哈希、token hash 和会话清理。
- `app/api/routes.py`：登出与 HTTP 409 写冲突响应。
- `app/api/schemas.py`：历史写入版本字段。
- `app/web/app.js`：版本回写、冲突提示、版本化删除、服务端登出。

### 仍需在公开部署前完成

- HTTPS（加密传输）、反向代理限流、CSRF（跨站请求伪造）策略与安全响应头；
- 登录失败限流/审计、邮箱验证与找回密码；
- Redis/数据库级分布式 Job（任务）队列、备份、监控与迁移工具；
- 导入仓库与异步 Job（任务）的持久化所有权校验和定期清理。

## 2026-07-24 — 仓库与异步任务所有权边界

- 导入仓库记录绑定 `owner_id`（用户标识）；接口返回不再包含服务器 `repo_path`（真实路径）。
- 前端调查请求只提交 `repo_id`（仓库标识）；后端按当前登录用户解析真实路径，越权或不存在统一返回 404。
- Investigation Job（调查任务）绑定创建用户；普通任务状态查询要求 Bearer Token（持有者令牌）所有权匹配。
- SSE（服务端事件流）使用每个 Job 的不可猜测 `stream_key`（流密钥），避免仅凭 job_id（任务标识）订阅结果。

## 2026-07-24 — 答案诊断与用户澄清

- 前端每次调查回答展示 Evidence（证据）数量、停止原因、未覆盖 Claim（断言）数量、已使用计划及最多 12 个证据文件行号。
- 诊断信息使用可展开面板，避免在正常回答中堆叠底层执行细节。
- 当没有 Evidence（证据）或因 `NO_PENDING`（没有待执行任务）停止时，前端主动引导用户补充类/函数名、目标文件或调用链起止点；该澄清消息也进入当前用户的独立历史。
